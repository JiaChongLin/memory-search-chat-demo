# 架构说明

## 设计目标

这个项目的架构目标不是追求抽象层次最多，而是满足下面几件事：

- 让聊天主流程清晰可读
- 让记忆、搜索、模型调用的责任边界明确
- 让前后端本地联调足够直接
- 让项目保持 demo 体量，不演化成通用 Agent 基础设施

## 总体结构图

```text
Browser
  |
  v
Frontend (HTML / CSS / JavaScript)
  |
  v
FastAPI
  |
  v
API Router (/health, /api/chat)
  |
  v
ChatService
  |------> MemoryService ------> SQLite
  |------> SearchService ------> Web Search
  `------> LLMService ---------> DashScope / Fallback
```

## 各模块职责

### Frontend

前端是一个纯静态页面，职责包括：

- 展示聊天界面
- 调用后端 API
- 缓存前端侧的 `session_id`
- 展示后端返回的摘要、搜索来源和运行状态

它不承担业务编排逻辑，也不做本地模型处理。

### API

API 层主要由 FastAPI 路由组成，职责包括：

- 提供健康检查接口
- 提供聊天接口
- 校验请求结构
- 统一错误返回格式

API 层本身尽量保持薄，只接收请求并调用 service。

### ChatService

ChatService 是当前后端主流程的编排层，职责包括：

- 确定 `session_id`
- 获取最近消息和摘要
- 判断是否需要搜索
- 调用搜索服务
- 调用模型服务
- 写入本轮消息
- 返回统一响应结构

它是“把多个能力串起来”的中心，但尽量不把细节逻辑都塞进去。

### MemoryService

MemoryService 负责会话记忆，职责包括：

- 查询最近消息
- 查询当前摘要
- 追加新一轮用户和助手消息
- 在条件满足时生成简化摘要
- 将摘要写回数据库

当前的摘要不是独立模型生成，而是对较早消息做简单压缩。

### SearchService

SearchService 负责搜索相关逻辑，职责包括：

- 判断当前用户问题是否应触发搜索
- 发起搜索请求
- 解析和整理搜索结果
- 返回统一的搜索结果结构

当前实现面向 demo，优先保证能演示，稳定性和复杂策略都比较有限。

### LLMService

LLMService 负责模型调用，职责包括：

- 组织 system prompt
- 注入会话摘要
- 注入搜索上下文
- 调用真实模型接口
- 在模型不可用时返回降级回复

它不负责数据库读写，也不负责决定是否搜索。

### Database

数据库当前采用 SQLite，职责包括：

- 存储会话
- 存储消息
- 存储会话摘要

当前适合本地单机 demo，不适合作为复杂生产环境方案。

## 一次聊天请求的处理流程

一次 `POST /api/chat` 请求的大致流程如下：

1. 前端提交 `message`，可选提交 `session_id`。
2. API 层完成参数校验，并交给 ChatService。
3. ChatService 确定使用已有会话还是创建新会话。
4. MemoryService 读取该会话最近消息。
5. MemoryService 读取该会话已有摘要。
6. SearchService 判断消息是否命中实时问题关键词。
7. 如果命中，则尝试执行搜索并返回结果。
8. LLMService 根据用户输入、历史消息、摘要和搜索结果组织 prompt。
9. LLMService 调用真实模型。
10. 如果真实模型调用失败，则返回本地降级回复。
11. MemoryService 将本轮用户消息和助手回复写入数据库。
12. MemoryService 视消息数量情况更新摘要。
13. ChatService 组装响应并返回给前端。

## 数据流

可以把当前数据流理解为三段：

### 输入流

- 用户在前端输入消息
- 前端把消息和 `session_id` 发给后端

### 编排流

- 后端读取数据库中的消息和摘要
- 后端视情况获取搜索结果
- 后端组织模型上下文
- 后端产出回复

### 持久化流

- 用户消息写入数据库
- 助手回复写入数据库
- 摘要更新写入数据库

最终前端拿到的是一个面向展示的响应对象，而不是数据库原始结构。

## 当前设计取舍

当前设计有几处明确取舍：

### 1. 选择简单分层，而不是高度抽象

好处是容易读、容易改；代价是模块能力还比较朴素。

### 2. 选择 SQLite

好处是本地开箱即用；代价是不适合复杂并发和迁移场景。

### 3. 选择关键词触发搜索

好处是实现成本低；代价是触发判断不够智能。

### 4. 选择本地降级回复

好处是模型出问题时，整条链路仍然可观察；代价是降级回复不是正式回答。

### 5. 摘要使用规则压缩，而不是独立模型摘要

好处是简单、稳定、少依赖；代价是摘要质量有限。

## 后续可演进方向

如果继续演进，可以优先考虑这些方向：

- 增加会话查询接口，便于调试和回放
- 将搜索触发策略从关键词扩展为更可控的规则或分类逻辑
- 改进摘要策略，例如引入更稳定的摘要更新规则
- 为搜索和模型调用增加更细的日志
- 为数据库引入迁移工具，而不是长期依赖 `create_all`
- 在不改变 demo 定位的前提下，逐步增强可观察性

需要注意的是，后续演进仍应保持项目边界：它应该继续是一个应用 demo，而不是演变成通用 Agent 平台。
