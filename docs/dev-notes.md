# Dev Notes

## 本阶段实现约定

### 1. 权限规则集中在 MemoryService

本阶段把聊天上下文访问边界集中放在 `MemoryService`，而不是散落到 `ChatService`。

当前职责划分：

- `ChatService`
  负责编排
- `MemoryService`
  负责上下文解析和权限收紧
- `LLMService`
  负责把解析后的上下文交给模型

这样下一阶段继续接 allowlist 或更细粒度规则时，会更容易扩展。

### 2. 项目层决定默认边界，会话层只能收紧

当前实际生效的优先级：

1. 先看当前会话是否 private
2. 再看当前会话所属项目的 `scope_mode`
3. 跨项目时再检查 `is_isolated` 边界
4. 最后过滤候选会话的 `status` / `is_private`

也就是说：

- 项目层负责给出默认范围
- 会话层不会放宽项目层范围，只会把范围收紧

### 3. 当前只读取其他会话摘要

其他会话当前不会把完整原始消息拼进模型上下文，只会带入摘要。

这么做的原因：

- 保持 token 成本更可控
- 降低误把过多历史塞进 prompt 的风险
- 让权限边界先落稳，再做更复杂的召回策略

### 4. archived 会话当前直接排除

需求里允许“archived 不优先使用”，当前实现选择更简单的策略：

- archived 会话不纳入上下文检索
- 文档中明确这一点

这样能减少阶段 2 的规则复杂度。

### 5. isolated 是边界，不是无限通行证

`scope_mode=global` 只表示“当前会话尽量读更多允许访问的历史”，并不绕过：

- `is_private`
- `is_isolated`
- `deleted`
- `archived`

这一点在测试里已经覆盖。

### 6. allowlist 仍未实现

当前没有 allowlist，也没有显式授权列表。

因此本阶段的权限系统仍然是：

- 项目层默认边界
- 会话层收紧规则
- 生命周期过滤

如果下一阶段要继续做更细粒度授权，建议优先在 `MemoryService.resolve_context()` 的候选过滤阶段扩展。

## 后续开发建议

如果继续推进下一阶段，建议优先检查：

- 是否需要把“项目外全局摘要”定义得更严格
- 是否需要单独区分“无项目会话”和“全局可访问会话”
- 是否要为 archived 引入低优先级而不是完全排除
- 是否要为 allowlist 增加显式白名单表或关联表
