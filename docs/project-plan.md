# 项目计划

## 项目概述

`memory-search-chat-demo` 是一个公开学习型 Web chat demo。

它的目标不是做通用 Agent 平台，也不是做 SDK 或通用工作流基础设施，而是实现一个最小可运行的聊天应用闭环，帮助读者理解下面几件事如何组合在一起：

- Web 聊天界面
- 多轮上下文
- 基础记忆
- 联网搜索增强
- 大模型回复生成

这个项目更接近“一个小型应用示例”，而不是“一个可复用平台产品”。

## 当前目标

当前阶段的目标很明确：

1. 提供一个可以本地直接跑起来的前后端 demo。
2. 让聊天、记忆、搜索、模型调用几条链路能串起来。
3. 在模型不可用或网络受限时，接口仍能返回可观察的降级结果。
4. 保持代码结构清晰，便于继续阅读和扩展。

## 当前已实现

截至当前版本，项目已经具备以下能力：

- 后端基于 FastAPI，已经提供基础 API 和统一错误响应。
- 提供 `POST /api/chat` 聊天接口。
- 支持通过 `session_id` 复用同一会话，实现多轮上下文。
- 使用 SQLite 保存会话、消息和会话摘要。
- 记忆能力包含两部分：
  - 最近若干条消息作为短期上下文
  - 会话摘要作为较早历史的压缩表示
- 搜索能力已经接入一个可用于 demo 的实现：
  - 先用关键词规则判断是否需要搜索
  - 命中后尝试获取网页搜索结果
- 大模型调用已经接入 DashScope 兼容接口。
- 当模型调用失败、配置缺失或网络不可用时，后端会返回本地降级回复，而不是直接整条接口失败。
- 已提供一个纯静态前端 demo 页面，用于本地联调。

## MVP 边界

这个项目的 MVP 边界比较明确。

MVP 内：

- 一个能发消息、拿回复的 Web 页面
- 一个能处理会话的聊天 API
- 基础记忆
- 简单搜索触发与搜索结果回传
- 基础数据库落盘
- 最小测试覆盖

MVP 外：

- 用户系统
- 权限体系
- 多租户
- 复杂的长期记忆管理
- 向量数据库
- 完整 RAG
- 工具调用编排
- 插件系统
- 通用 Agent 抽象层
- 生产级可观测性和高可用设计

## 技术路线

当前采用的是比较朴素的分层方案：

- Frontend：纯 HTML / CSS / JavaScript
- API：FastAPI 路由层
- Service：聊天编排、记忆、搜索、模型调用分开实现
- Database：SQLite + SQLAlchemy

核心路线是：

1. 前端负责输入输出和状态展示。
2. API 层负责参数接收和错误边界。
3. ChatService 负责把记忆、搜索和模型调用串起来。
4. MemoryService 负责最近消息和摘要。
5. SearchService 负责触发判断和搜索结果整理。
6. LLMService 负责组织上下文和调用模型。
7. Database 负责持久化会话数据。

这种路线的优点是简单直接，适合 demo；缺点是抽象层不深，不适合直接作为大规模系统基础。

## 下一阶段计划

下一阶段更值得做的工作包括：

- 增加会话查询或调试辅助接口，便于查看历史消息和摘要。
- 继续优化前端中文显示与交互体验。
- 调整搜索触发策略，减少误触发或漏触发。
- 增加更稳定的搜索结果解析与异常处理。
- 继续补充测试，覆盖更多搜索和异常场景。
- 适度补充日志，方便定位本地联调问题。

## 风险与限制

当前版本有这些明确限制：

- 搜索实现是 demo 级方案，不保证稳定性和长期兼容性。
- 搜索触发依赖关键词规则，判断能力有限。
- 摘要逻辑是简化实现，不是高质量语义摘要。
- LLM 调用当前主要面向单一提供方。
- 使用 SQLite 和 `create_all`，适合本地 demo，不适合复杂迁移场景。
- 当前没有提供会话管理页面或会话列表 API。
- 当前不是生产级高并发设计。

## 成功标准

这个项目在当前阶段是否达标，可以用下面几条判断：

1. 前端页面可以正常连接后端并发起聊天请求。
2. 首次请求会创建 `session_id`，后续请求可以复用。
3. 多轮对话后可以看到摘要开始生成。
4. 实时问题会触发搜索逻辑，并在可能时返回来源列表。
5. 模型可用时走真实模型；模型不可用时返回可观察的降级结果。
6. 代码结构对学习者来说足够清晰，能够支撑继续改造。
